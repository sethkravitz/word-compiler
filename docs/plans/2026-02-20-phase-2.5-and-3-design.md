# Phase 2.5 + Phase 3 Design

> **Historical document.** This design plan was written during active development and references AI model consultations used during the design process. The implementation may have diverged — see `docs/architecture/` for current documentation.

## Status Quo

Phase 2 built all the core logic modules but left them disconnected. The compiler, auditor, IR extractor, metrics, gates, and bootstrap all work in isolation (505 tests prove it). The backend has a full SQLite DB with 40+ REST endpoints. But the UI is client-side only — reload the page and everything is gone.

The generation loop works (DraftingDesk streams LLM tokens, runs audit), but nothing persists. Phase 3's revision learner needs persistence and a working writing loop to learn from. So we finish Phase 2 integration first.

### Sprint A: Persistence Layer

**Architecture Decision: Server-First ("Approach C+")**

Consulted GPT-5 (for simplicity) vs GPT-5-Codex (for robustness). Consensus: server-first wins for a local single-user desktop tool where SQLite latency is ~1ms. No IndexedDB, no offline sync, no optimistic updates — unnecessary complexity when the "server" is localhost.

Enhancements over pure server-first:
- Debounce text content PATCHes (500ms idle) to avoid save storms during editing
- Streaming tokens update store directly; persist chunk on generation completion
- UI-only state (tab selection, modal open/closed) does not persist

**Implementation:**

1. **Create `src/app/store/api.ts`** — ApiClient interface + fetch implementation
   ```typescript
   interface ApiClient {
     getProject(id: string): Promise<Project>;
     createProject(data: Partial<Project>): Promise<Project>;
     listProjects(): Promise<Project[]>;
     saveBible(bible: Bible): Promise<Bible>;
     saveChunk(chunk: Chunk): Promise<Chunk>;
     // ... one method per API endpoint
   }
   ```
   Production impl uses `fetch()`. Tests inject a mock.

2. **Wire store mutations** — Each mutation calls API first, then updates state from response:
   ```typescript
   async setBible(bible: Bible) {
     const saved = await this.api.saveBible(bible);
     this.bible = saved;  // update from server response
     this.error = null;
   }
   ```

3. **Debounce text edits** — Component-level debounce for editedText PATCHes. Store updates immediately (for responsive UI), API call fires after 500ms idle.

4. **Startup flow** — App mounts → fetch project list → if one project, load it → call `store.loadFromServer()`. If no projects, show create/import screen.

5. **Auto-persist after generation** — `generateChunk()` already creates chunks in store. Add `await api.saveChunk(chunk)` after stream completes.

6. **Persist audit flags, IR, compilation logs** — After each audit run, save flags to API. After IR extraction, save to API. Compilation logs saved on each compile.

7. **Tests** — Create `MockApiClient` that returns canned data. Inject into store for component tests. Existing mock stores remain unchanged for tests that don't need persistence.

### 3A: Diff Analysis Pipeline

**New module: `src/learner/diff.ts`**

Algorithm: Sentence-level Myers diff with secondary word-level classification.

Why sentence-level: Word-level diffs are too noisy for creative text (a rewritten sentence shows as many word changes). Paragraph-level misses substitutions. Sentence-level captures the right granularity for editorial decisions.

**Sentence segmentation:**
- Split on `.` `!` `?` followed by whitespace or newline
- Dialogue-aware: don't split on periods inside quotation marks
- Preserve paragraph boundaries as segment delimiters

**Diff process:**
1. Segment `generatedText` and `editedText` into sentences
2. Run Myers diff at sentence level (matched, added, deleted, modified)
3. For "modified" sentences (edit distance < 60%), run word-level diff to classify

**Edit taxonomy (4 categories, 8 sub-types):**

| Category | Sub-type | Signal | Bible Action |
|----------|----------|--------|--------------|
| DELETION | CUT_FILLER | Short phrase removed | Kill list addition |
| DELETION | CUT_PASSAGE | Full sentence/paragraph removed | Verbosity calibration |
| SUBSTITUTION | TONE_SHIFT | Adjective/adverb swap | Style guide refinement |
| SUBSTITUTION | DIALOGUE_VOICE | Dialogue rewritten | Voice fingerprint update |
| SUBSTITUTION | SHOW_DONT_TELL | Abstract to concrete | Exemplar addition |
| ADDITION | BEAT_ADDED | Physical action inserted near dialogue | Character voice note |
| ADDITION | SENSORY_ADDED | Sensory detail inserted | Sensory palette update |
| RESTRUCTURE | REORDER | Sentences moved within paragraph | Pacing preference |

**Sub-type precedence:** When an edit matches multiple types, RESTRUCTURE wins (it's structural), then SUBSTITUTION (semantic change), then DELETION/ADDITION (content change).

**New DB tables:**

```sql
CREATE TABLE edit_patterns (
  id TEXT PRIMARY KEY,
  chunk_id TEXT NOT NULL,
  scene_id TEXT NOT NULL,
  project_id TEXT NOT NULL,
  edit_type TEXT NOT NULL,       -- DELETION, SUBSTITUTION, ADDITION, RESTRUCTURE
  sub_type TEXT NOT NULL,        -- CUT_FILLER, TONE_SHIFT, etc.
  original_text TEXT NOT NULL,
  edited_text TEXT NOT NULL,
  context TEXT,                  -- surrounding sentences for display
  created_at TEXT NOT NULL,
  FOREIGN KEY (chunk_id) REFERENCES chunks(id)
);

CREATE TABLE learned_patterns (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  pattern_type TEXT NOT NULL,    -- maps to sub_type
  pattern_data TEXT NOT NULL,    -- JSON: { phrases: [...], examples: [...] }
  occurrences INTEGER NOT NULL,
  confidence REAL NOT NULL,      -- Wilson interval lower bound
  status TEXT NOT NULL DEFAULT 'proposed',  -- proposed, accepted, rejected, expired
  proposed_action TEXT,          -- JSON: { target: "kill_list", value: "a sense of" }
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (project_id) REFERENCES projects(id)
);

CREATE TABLE profile_adjustments (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  parameter TEXT NOT NULL,       -- temperature, ring1_budget, chunk_target, etc.
  old_value REAL NOT NULL,
  new_value REAL NOT NULL,
  evidence TEXT NOT NULL,        -- JSON: { edit_ratio: 0.45, chunks_analyzed: 12 }
  status TEXT NOT NULL DEFAULT 'proposed',
  created_at TEXT NOT NULL,
  FOREIGN KEY (project_id) REFERENCES projects(id)
);
```

**Trigger:** Auto-classify when chunk is saved with `editedText !== null && editedText !== generatedText`.

### 3B: Pattern Accumulation

**New module: `src/learner/patterns.ts`**

- Group `edit_patterns` by `(sub_type, normalized_pattern)` where `normalized_pattern` is the lowercase deleted/substituted text
- **Wilson interval** for confidence scoring (accounts for sample size):
  ```
  confidence = (p + z^2/(2n) - z*sqrt(p*(1-p)/n + z^2/(4*n^2))) / (1 + z^2/n)
  where p = successes/n, z = 1.96 (95% CI)
  ```
- **Promotion threshold:** >= 5 occurrences AND Wilson lower bound >= 0.60
- **Evidence decay:** 30-scene sliding window. Patterns older than 30 scenes carry 50% weight. This prevents early project habits from permanently locking in.
- **Pattern lifecycle:** `edit_patterns` (raw) -> accumulate -> `learned_patterns` (proposed) -> user accepts/rejects -> applied to bible or expired

### 3C: Bible Update Proposals

**UI: "Learner Suggestions" panel** (new tab or sidebar section)

Each proposal shows:
- **What:** "Add 'a sense of' to kill list" or "Update Marcus's voice notes: shorter dialogue"
- **Evidence:** 3-5 example edits with before/after snippets and scene context
- **Confidence:** percentage bar + "Based on 7 edits across 4 scenes"
- **Actions:** Accept (auto-applies to bible) | Reject (marks pattern as rejected, won't re-propose) | Modify (opens editor to adjust before applying)

Proposal types mapped to bible sections:
- CUT_FILLER -> `killList[]`
- DIALOGUE_VOICE -> `characters[].voiceNotes`
- SHOW_DONT_TELL -> `suggestedTone.exemplars[]`
- SENSORY_ADDED -> `locations[].sensoryPalette`
- BEAT_ADDED -> `characters[].emotionPhysicality`
- TONE_SHIFT -> `suggestedTone.metaphoricDomains` or `prohibitedDomains`

### 3D: Compilation Profile Auto-Tuning

**Metrics tracked per chunk:**
- Edit distance ratio: `levenshtein(generated, edited) / len(generated)`
- Exclude RESTRUCTURE edits (structural, not quality signal)
- Track per-ring correlation: which ring sections appear in heavily-edited vs lightly-edited chunks

**Tuning proposals (require >= 10 chunks across >= 3 scenes):**
- High edit ratio (> 40%) -> propose lower temperature
- Ring 1 sections frequently cut -> propose tighter Ring 1 budget
- Consistently short final text vs target -> propose shorter chunk target
- Low edit ratio (< 10%) -> suggest current settings are working well (positive signal)

## Expert Analysis Notes

### Persistence Strategy
Consulted GPT-5 (simplicity advocate) and GPT-5-Codex (robustness advocate). GPT-5 scored Approach C (server-first) at 8/10. GPT-5-Codex scored Approach B (sync layer) at 7/10. Synthesis: server-first wins because offline sync adds complexity with no user benefit for a local desktop tool. The "centralized persistence" argument from Approach B is achievable via an injectable ApiClient interface.

### Revision Learner
GPT-5-Codex expert analysis raised key refinements:
- Sentence segmentation in dialogue-heavy prose is brittle. Need dialogue-aware boundary detection.
- Wilson interval > simple ratio for confidence scoring (accounts for small sample sizes).
- Evidence decay prevents early-project habits from permanently locking in.
- Sub-type precedence rules needed when edits match multiple categories.
- Gate profile tuning on minimum data (>= 10 chunks) to avoid noisy early suggestions.
